# Wiki Query（知识库查询）

基于 Wiki 页面内容回答用户问题，通过搜索、排序、综合三个阶段给出精准回答。

## 边界

基于 Wiki 已有内容回答问题。不摄入新资料（那是 `/ingest`），不做健康检查（那是 `/lint`）。

如果用户的问题需要新资料才能回答，提示用户提供资料并使用 `/ingest`。

## 判断是否需要查询

不是所有问题都需要走完整查询流程。快速判断：

- **需要查询**：问题涉及知识库中的具体概念、实体、主题，需要综合多个页面回答
- **直接回答**：一般性编程问题、数学公式、常识问题——这些不需要查询知识库
- **不确定**：先快速搜索，有结果就走查询流程，没有就告知用户知识库中暂无相关内容

## 搜索流程

**优先使用 Obsidian CLI**（`/obsidian-cli` skill）读取和搜索 Wiki 页面。Obsidian CLI 能直接与 Obsidian vault 交互，获取最新的页面内容和元数据。

### 步骤 1：qmd 搜索

使用 qmd 混合搜索获取匹配页面：

**优先使用 MCP 工具**：调用 `query` MCP 工具进行混合搜索。

**回退到 CLI**：
```bash
qmd-node query "用户的查询内容"
```

### 步骤 2：读取摘要

**优先使用 Obsidian CLI** 读取页面内容。对每个搜索结果页面，读取摘要段落（frontmatter 后到第一个 `##` 之间的内容）。

摘要段落的目的是快速判断页面相关性，不需要读全文。

### 步骤 3：聚合排序

按以下规则对匹配页面排序：

1. **同页面多段落命中** → 排名提升（说明该页面覆盖面广）
2. **概念页面 > 来源摘要页面** → 概念页更精炼，优先展示
3. **被更多页面引用** → 微调提升（说明该页面是核心知识点）

### 步骤 4：深入阅读

仅对排序后的 **top 3 页面**读取全文。这能控制 token 消耗。

如果 top 3 不够充分，可以扩展到 top 5，但要告知用户搜索范围扩大了。

### 步骤 5：综合回答

基于 Wiki 内容（而非原始资料）回答用户问题，遵循以下原则：

- 回答要综合多个相关页面的信息，不是复述单一页面
- 引用相关页面使用 `[[wikilink]]` 语法，方便用户在 Obsidian 中跳转
- 如果不同页面有互补视角，要体现出来
- 如果 Wiki 中的信息不足以完整回答，坦诚说明并建议补充相关资料

### 步骤 6：回填洞察

如果回答过程中产生了新的洞察（以下任一情况），创建综合页面：

**需要回填的情况**：
- 对比了多个概念/实体的异同
- 发现了页面间未被记录的关联
- 综合多个来源得出了新结论
- 回答本身就是一篇综述性的分析

**不需要回填的情况**：
- 简单的事实查询（"XX 是什么"）
- 直接复述单个页面的内容
- 一般性的解释和定义

回填时，**优先使用 Obsidian CLI** 创建页面和更新索引。按 CLAUDE.md 中综合页面的格式创建 `wiki/synthesis/` 页面，并同步更新 `wiki/index.md` 和 `wiki/log.md`。

## 搜索回退策略

如果搜索结果不足：

1. **qmd 结果 < 3 个页面** → 读取 `wiki/index.md` 手动定位相关页面
2. **index.md 也没有** → 提示用户知识库中暂无相关内容，建议：
   - 补充相关资料（使用 `/ingest`）
   - 或运行 `/lint` 检查是否有孤立页面未被索引

数字签名是提供身份认证、完整性和不可否认性的密码学技术，基于公私钥对实现：只有私钥持有者能生成签名，任何人都能用公钥验证。本页涵盖 RSA-PSS、ECDSA、EdDSA 等签名方案、EUF-CMA 安全模型、签名安全强度的木桶短板原则以及典型应用场景。

# 数字签名

## 定义

数字签名（Digital Signature）是提供身份认证（authentication）、完整性（integrity）和不可否认性（non-repudiation）的密码学技术。只有私钥持有者能生成签名，任何人都能用公钥验证签名真伪。与 HMAC 的关键区别在于：HMAC 使用共享密钥，无法向第三方证明签名者身份；数字签名使用公私钥对，可向任何第三方证明。

## 关键要点

### 安全模型：EUF-CMA

数字签名的安全定义为 **EUF-CMA**（Existential Unforgeability under Chosen Message Attack）：攻击者可以获取任意消息的签名，但仍无法为任何新消息伪造合法签名。签名方案由三元组组成：密钥生成（G）、签名（S）、验证（V）。

### 签名安全强度：木桶短板原则

签名的安全强度由摘要算法和公钥算法共同决定，取最弱者。例如 RSA-2048（112 位安全）+ SHA-256（128 位安全）= 整体 112 位安全。

推荐组合：
- **入门级**：RSA 2048 + SHA-256（112 位）
- **推荐级**：ECDSA P-256 + SHA-256（128 位）
- **高安全**：ECDSA P-384 + SHA-384（192 位）

### 主要签名算法

#### DSA（Digital Signature Algorithm）

1991 年 NIST 发布，基于离散对数问题。2048 位密钥提供 112 位安全。签名由 (R, S) 两个整数组成。随机数 K 的安全性至关重要——Sony PS3 破解事件就是因为 K 值硬编码。确定性 DSA（RFC 6979）从私钥和消息哈希推导 K，消除随机数依赖。

#### ECDSA（Elliptic Curve DSA）

DSA 在椭圆曲线上的实现，256 位密钥达到 128 位安全（vs DSA 需要 3072 位）。广泛兼容（TLS、JWT、代码签名）。签名非确定性（每次不同），依赖随机 nonce。nonce 重用会导致私钥泄露（致命）。

#### EdDSA（Edwards Curve DSA）

基于 Edwards 曲线和 Schnorr 签名，当前新项目首选。

- **Ed25519**：~128 位安全，64 字节固定签名，确定性（无需随机数），抗侧信道
- **Ed448**：~224 位安全，114 字节签名

三大优势：
1. 确定性签名——nonce 从私钥和消息确定性推导，消除随机数安全隐患
2. 天然抗侧信道——Edwards 曲线运算天然常量时间
3. 签名恒定长度——不像 DSA/ECDSA 因 DER 编码有 1 字节差异

#### RSA 签名

- **PKCS#1 v1.5**：确定性，极广泛兼容，但实现校验不严格时可被伪造
- **RSA-PSS**：引入随机盐值，有可证明安全性，新项目推荐

#### SM2 国密签名

中国密码管理局标准，基于 sm2p256v1 椭圆曲线 + SM3 摘要。独特之处在于签名前的消息预处理（用户身份前缀 ZA），给每个签名者打上"身份烙印"。合规场景（政务、金融、电子签章）必需。

### Hash-and-Sign 范式

签名不对原始消息操作，而是先哈希再签名：S'(sk, m) = S(sk, H(m))。安全性依赖哈希的碰撞抗性。裸 RSA 签名（无哈希无填充）存在无消息伪造漏洞——攻击者先选随机签名，再反算出对应消息。

### 签名延展性攻击

ECDSA 签名 (r, s) 中，(r, -s mod n) 也是合法签名。攻击者无需私钥就能生成不同的合法签名。2014 年 MtGox 事件中攻击者利用此特性修改交易签名，导致系统误判重复发款。缓解措施：Bitcoin SegWit、Ed25519 规范要求 s < n/2。

### Schnorr 签名与聚合

EdDSA 基于 Schnorr 签名。Schnorr 的线性性允许多方签名聚合：多个签名者的签名分量可直接相加，合并为一个与单个签名大小相同的聚合签名。MuSig 协议安全地实现了多方聚合。比特币 Taproot（2021）采用 Schnorr 签名正是为了聚合能力。

### Java API 使用模式

无论哪种算法，`java.security.Signature` 的使用都遵循三步：初始化（initSign/initVerify）→ 传入数据（update）→ 签名/验签（sign/verify）。

```java
// 签名
Signature sig = Signature.getInstance("SHA256withECDSA", "BC");
sig.initSign(privateKey);
sig.update(data);
byte[] signatureBytes = sig.sign();

// 验签
sig.initVerify(publicKey);
sig.update(data);
boolean valid = sig.verify(signatureBytes);
```

## 与其他概念的关系

- [[密码学]]：数字签名是密码学的核心子领域，与非对称加密共用公私钥对但方向相反（加密：公钥加密→私钥解密；签名：私钥签名→公钥验证）
- [[TLS协议]]：TLS 握手使用数字签名（CertificateVerify）验证服务端身份

## 选型建议

| 场景 | 推荐算法 |
|------|---------|
| 新项目（通用） | **Ed25519**（确定性、抗侧信道、签名紧凑） |
| 广泛兼容性需求 | ECDSA P-256 或 RSA PKCS#1 v1.5 |
| 理论安全性 | RSA-PSS |
| 中国合规 | SM2 |
| 嵌入式/物联网 | Ed25519（小密钥、小签名、无随机数依赖） |
| 后量子 | ML-DSA（见 [[后量子密码]]） |
